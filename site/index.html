01/ how it works

It's just fork() — but for a running LLM.

01

Snapshot a live session.

Weights, KV cache, scheduler state, prefix-hash table — captured in place without tearing down the engine. Pinned memory, pipelined DMA, O_DIRECT writes. The session stays hot.

handle = thaw.fork(llm)
02

Hydrate N children.

Each child inherits the fork point and starts generating at the exact token you forked at. No prefill. No re-reading of the prompt. Same process, or farmed out to pre-warmed workers — your call.

handle.completions(n=4)
03

Diverge. Rank. Keep one.

RL rollouts, tool-call fan-out, tree search, best-of-N — each branch runs to completion, you score, you keep the winners. Sub-second per round after a one-time warmup.

pool.fork_completions(...)

03/ audience

You probably need this if:

_ RL post-training

You pay prefill N times per rollout.

PPO, GRPO, rejection sampling — every retry reprocesses the same prompt prefix. With thaw, prefill happens once; the 16 rollouts fork off the same KV cache.

Arithmetic ROI on re-prefill cost
_ Coding-agent companies

Your agent branches five times per turn.

Tool-call fan-out, self-consistency, best-of-N code gen. Every branch is the same 8k-token context with a different decoding path. Don't reprocess prefill each time.

Fan-out without paying prefill
_ Agent framework authors

You're building LangGraph, CrewAI, Autogen…

Tree-of-thought, MCTS over agents, parallel reviewer fan-out — all of it wants a fork primitive at the runtime layer, not a cache proxy in front.

A real primitive, not a proxy
